📦 MODULE 1 — UNIT 1: INTRODUCTION TO BIG DATA

1. Define Data & Characteristics of Data

Definition
Data is a collection of raw, unprocessed facts, figures, symbols, or values that are meaningless on their own but become meaningful when organized, processed, and interpreted as information.

Characteristics of Data

  • Accuracy: Data should be correct and free from errors.
  • Completeness: Data should not have missing values.
  • Consistency: Data should be uniform across different systems.
  • Timeliness: Data should be up-to-date and available when needed.
  • Relevance: Data collected must be relevant to the problem.
  • Accessibility: Data should be easily retrievable and usable.
  • Integrity: Data should be protected from unauthorized modification.

4. Working with Unstructured Data

Theory
Unstructured Data is data that does not conform to a predefined data model and cannot be easily stored in traditional relational databases. It has no consistent structure or format.

Challenges in Handling Unstructured Data

  • Storage: Requires massive, distributed storage systems (HDFS, S3).
  • Processing: Cannot use standard SQL queries — needs NLP, ML, image recognition.
  • Indexing: Difficult to index and search efficiently.
  • Quality: Noise, redundancy, and irrelevant data must be cleaned.
  • Integration: Hard to combine with structured data.
  • Security: Controlling access to unstructured data is complex.

Techniques Used

  • Natural Language Processing (NLP) for text
  • Computer Vision for images/videos
  • Speech Recognition for audio
  • Apache Tika for metadata extraction
  • Hadoop / Spark for distributed processing

6. Define Big Data

Definition
Big Data refers to extremely large datasets that cannot be captured, stored, processed, or analyzed using traditional database management tools within an acceptable time frame. It is characterized by high Volume, Velocity, and Variety.

Big Data is defined by Gartner as: "Big data is high-volume, high-velocity, and/or high-variety information assets that demand cost-effective, innovative forms of information processing that enable enhanced insight, decision making, and process automation."

📦 MODULE 1 — UNIT 2: INTRODUCTION TO HADOOP

20. What is Hadoop?

Apache Hadoop is an open-source framework for distributed storage and processing of large datasets using a cluster of commodity hardware. It is based on the Google File System (GFS) and Google's MapReduce programming model.

Key Features

  • Fault Tolerant: Automatically replicates data across nodes (default 3 replicas).
  • Scalable: Add more nodes to handle more data (horizontal scaling).
  • Cost-Effective: Runs on cheap commodity hardware.
  • Flexible: Handles all types of data — structured, semi-structured, unstructured.
  • Open Source: Free to use with a large community.

Core Components of Hadoop

  • HDFS — Hadoop Distributed File System (storage)
  • MapReduce — Processing engine
  • YARN — Resource management (Hadoop 2+)
  • Hadoop Common — Shared utilities and libraries

21. Need of Hadoop

Very Common
  • Traditional RDBMS cannot handle petabytes of unstructured data.
  • Vertical scaling of traditional databases is very expensive.
  • Traditional systems cannot process data in parallel efficiently.
  • Hadoop allows storing raw data first and processing later (Schema-on-Read).
  • Internet-scale companies (Google, Yahoo, Facebook) needed a way to process billions of web pages.
  • IoT, social media, and sensor data generate data at incredible speeds requiring a scalable system.

22. Limitations of RDBMS

Very Important
  • Scalability: RDBMS supports only vertical scaling (adding more RAM/CPU to one machine). Very expensive.
  • Unstructured Data: Cannot efficiently store images, videos, text, audio.
  • Speed Limitations: With large datasets, query performance degrades significantly.
  • Cost: Enterprise RDBMS (Oracle, SQL Server) licenses are very expensive.
  • Schema Rigidity: Schema must be defined before data is stored.
  • Data Volume: Not designed to handle petabytes/exabytes of data.
  • Real-time Streaming: Poor support for real-time ingestion of streaming data.

24. Distributed Computing Challenges

  • Network Failures: Nodes can lose connectivity; system must handle this gracefully.
  • Partial Failures: Some nodes may fail while others are running.
  • Data Consistency: Ensuring all nodes see the same data (CAP theorem).
  • Fault Tolerance: System must continue operating even when nodes fail.
  • Load Balancing: Work must be evenly distributed to prevent hot spots.
  • Synchronization: Coordinating distributed tasks without race conditions (ZooKeeper).
  • Data Locality: Moving computation to data rather than data to computation.
  • Security: Securing communication between nodes in the cluster.

25. History of Hadoop

Key People: Doug Cutting (creator) and Mike Cafarella (co-creator). Hadoop is named after Doug Cutting's son's toy yellow elephant.
  • 2003: Google published the Google File System (GFS) paper.
  • 2004: Google published the MapReduce paper.
  • 2004–2005: Doug Cutting and Mike Cafarella implemented GFS and MapReduce concepts in a project called Nutch (a web crawler).
  • 2006: Yahoo hired Doug Cutting. The distributed file system and MapReduce were split from Nutch and named Hadoop.
  • 2008: Hadoop became an Apache top-level project.
  • 2011: Hadoop 1.0 released.
  • 2013: Hadoop 2.0 released with YARN (Yet Another Resource Negotiator).
  • 2017: Hadoop 3.0 released with erasure coding, improved performance.

26. Hadoop Architecture

🔥 MUST PREPARE FULL DIAGRAM
Hadoop Architecture (Master-Slave Model)

How it Works

  • Client submits jobs to the Master Node.
  • NameNode manages HDFS metadata (where data blocks are located).
  • ResourceManager manages cluster resources and schedules YARN jobs.
  • DataNodes store actual data blocks and report to NameNode via heartbeat.
  • NodeManagers execute MapReduce tasks and report to ResourceManager.

27. HDFS Architecture (Hadoop Distributed File System)

🔥 Very Very Important
HDFS is the primary storage system of Hadoop. It stores large files by splitting them into fixed-size blocks (default 128 MB) and distributing them across multiple DataNodes, with each block replicated on 3 nodes by default.
HDFS Architecture

Key HDFS Concepts

ConceptDescription
Block SizeDefault 128 MB (Hadoop 2+). Large files are split into blocks and distributed.
Replication FactorDefault 3 — each block is stored on 3 different DataNodes for fault tolerance.
NameNodeMaster — stores metadata (filenames, block locations, permissions). Does not store actual data.
DataNodeSlave — stores actual data blocks. Sends heartbeat to NameNode every 3 seconds.
Secondary NameNodePeriodically merges FsImage and EditLog. NOT a backup NameNode.
HeartbeatDataNodes send heartbeat every 3 seconds to prove they're alive.
Block ReportDataNodes send a list of all blocks they store to the NameNode periodically.
Rack AwarenessHDFS places replicas on different racks to survive rack-level failures.

HDFS Read Operation

  1. Client contacts NameNode for block locations.
  2. NameNode returns list of DataNodes for each block.
  3. Client reads blocks directly from the nearest DataNode.

HDFS Write Operation

  1. Client notifies NameNode to create a file.
  2. NameNode allocates blocks and DataNode pipeline.
  3. Client writes data to first DataNode in the pipeline.
  4. First DataNode replicates to second, second to third.
  5. Acknowledgment sent back to client.

33. MapReduce Architecture

🔥 MUST DIAGRAM
MapReduce is a programming model and processing framework for distributed processing of large datasets across a cluster. It divides the task into two phases: Map (parallel processing) and Reduce (aggregation).
MapReduce Detailed Data Flow

34. Mapper, Reducer, Combiner, Partitioner

Mapper

The Mapper processes each input record (key-value pair) and produces zero or more intermediate key-value pairs. It runs in parallel across all data splits.

Signature: map(key, value) → list(key2, value2)

Example (Word Count): Input: ("line1", "hello world") → Output: [("hello",1), ("world",1)]

Reducer

The Reducer receives all values associated with a given intermediate key (after shuffle & sort) and produces the final output.

Signature: reduce(key, list(values)) → list(key3, value3)

Example: Input: ("hello", [1,1,1]) → Output: ("hello", 3)

Combiner (Mini-Reducer)

The Combiner is an optional local aggregation step that runs on the mapper's output before data is sent over the network to the reducer. It reduces network traffic.

Same interface as Reducer. Example: Instead of sending ("hello",1), ("hello",1), combiner sends ("hello",2).

Partitioner

The Partitioner determines which reducer receives a particular intermediate key. Default: HashPartitioner — uses hash(key) % numReducers.

Custom partitioners ensure all keys in the same group go to the same reducer.

36. Searching and Sorting using MapReduce

Sorting using MapReduce

MapReduce has built-in sorting in the Shuffle & Sort phase. The framework automatically sorts mapper outputs by key before sending to reducers. There are 3 types:

  • Default Sort: Keys are sorted lexicographically before each reducer receives data.
  • Secondary Sort: Sort by composite key (e.g., sort by year, then by temperature).
  • Total Order Sort (TeraSort): Globally sorted output across all reducers using a Partitioner that partitions ranges.

Searching using MapReduce

Pattern: Mapper filters records matching a search criterion. Reducer collects all matches.

Mapper:
  if value.contains(searchTerm):
    emit(key, value)

Reducer:
  emit all received matches

38. Matrix Multiplication using MapReduce

🔥 Very Common Long Question

Given matrices A (m×n) and B (n×p), compute C = A×B where C[i][j] = Σ A[i][k] * B[k][j]

Phase 1 — Map Phase

  • For each element A[i][k], emit key (i, j) for all j: emit((i,j), ("A", k, A[i][k]))
  • For each element B[k][j], emit key (i, j) for all i: emit((i,j), ("B", k, B[k][j]))

Phase 2 — Reduce Phase

For each key (i, j), collect all "A" values with index k and all "B" values with index k, then compute: C[i][j] = Σ A_values[k] * B_values[k]

// Example: A (2×2) × B (2×2)
// A = [[1,2],[3,4]], B = [[5,6],[7,8]]

// Mapper for A[0][0]=1, A[0][1]=2 (row 0 of A)
emit((0,0), (A,0,1))  // contributes to C[0][0]
emit((0,1), (A,0,1))  // contributes to C[0][1]
emit((0,0), (A,1,2))  // contributes to C[0][0]
emit((0,1), (A,1,2))  // contributes to C[0][1]

// Reducer for key (0,0):
// Gets: (A,0,1),(A,1,2),(B,0,5),(B,1,7)
// C[0][0] = 1*5 + 2*7 = 5+14 = 19

39. PageRank Algorithm using MapReduce

🔥 High Chance
PageRank is an algorithm by Google that assigns a numerical weight to each webpage to measure its importance based on the number and quality of links pointing to it.

PageRank Formula

PR(A) = (1-d) + d × [PR(T1)/C(T1) + PR(T2)/C(T2) + ... + PR(Tn)/C(Tn)]

  • d = damping factor (0.85)
  • PR(Ti) = PageRank of page Ti that links to A
  • C(Ti) = number of outbound links from Ti

MapReduce Implementation

Iterative Algorithm (run multiple times until convergence)

Mapper:

Input: (pageID, currentPR, [list of outlinks])
For each outlink L in outlinks:
    emit(L, currentPR / numOutlinks)
emit(pageID, list_of_outlinks)  // preserve graph structure

Reducer:

Input: (pageID, [list of PR contributions + outlink list])
newPR = 0
For each contribution c:
    newPR += c
newPR = (1 - 0.85) + 0.85 * newPR
emit(pageID, newPR, outlinks)

📦 MODULE 2: APACHE PIG

42. Introduction to Apache Pig

Apache Pig is a high-level data flow platform for processing large datasets on Hadoop. It uses a scripting language called Pig Latin that abstracts the complexity of MapReduce. Pig Latin scripts are automatically compiled into MapReduce jobs.

Why Pig?

  • MapReduce requires complex Java programs even for simple operations.
  • Pig Latin is much simpler — 10 lines of Pig can replace 200 lines of Java MapReduce.
  • Pig supports joins, filters, groupings, sorting — operations not native to MapReduce.

Key Features

  • Ease of Programming: SQL-like syntax for data pipelines.
  • Optimization: Automatic optimization of queries.
  • Extensibility: User-Defined Functions (UDFs) in Java, Python.
  • Handles all data formats.

43. Pig Architecture

🔥 Important Diagram
Apache Pig Architecture

Pig Architecture Components

  • Pig Latin: The scripting language users write.
  • Parser: Checks syntax and creates a Logical Plan (DAG).
  • Optimizer: Optimizes the Logical Plan (e.g., pushes filters down).
  • Compiler: Converts Logical Plan to Physical Plan, then to MapReduce jobs.
  • Execution Engine: Submits MR jobs to Hadoop cluster for execution.

44. Pig Latin Overview

Pig Latin is a dataflow language. Each statement takes a relation as input and produces a relation as output.

Basic Structure

-- Load data
A = LOAD 'hdfs://input' USING PigStorage(',') AS (name:chararray, age:int, salary:float);

-- Filter
B = FILTER A BY age > 30;

-- Group
C = GROUP B BY name;

-- Aggregate
D = FOREACH C GENERATE group, COUNT(B) AS count;

-- Store
STORE D INTO 'hdfs://output';

48. Word Count using Apache Pig

-- Word Count in Pig Latin

-- Step 1: Load the input file
lines = LOAD 'hdfs://input/data.txt' AS (line:chararray);

-- Step 2: Tokenize each line into words
words = FOREACH lines GENERATE FLATTEN(TOKENIZE(line)) AS word;

-- Step 3: Group by word
grouped = GROUP words BY word;

-- Step 4: Count each word
word_count = FOREACH grouped GENERATE group AS word, COUNT(words) AS count;

-- Step 5: Sort by count descending (optional)
sorted = ORDER word_count BY count DESC;

-- Step 6: Store result
STORE sorted INTO 'hdfs://output/wordcount';

📦 MODULE 2: APACHE HIVE

50. Introduction to Apache Hive

Apache Hive is a data warehouse infrastructure built on top of Hadoop. It provides a SQL-like query language called HiveQL (HQL) that allows users to query and analyze large datasets stored in HDFS as if they were in a relational database.

Key Features

  • SQL-like Interface: HQL is very similar to SQL — analysts can use Hive without MapReduce knowledge.
  • Scalable: Processes petabytes of data on Hadoop cluster.
  • Schema on Read: Schema is applied when data is read, not when stored.
  • Multiple File Formats: Supports TextFile, ORC, Parquet, Avro, SequenceFile.
  • Partitioning & Bucketing: For query optimization.
  • UDFs: Extensible with custom User-Defined Functions.

Who Uses Hive?

Facebook (original creator), Netflix, Amazon, LinkedIn — any organization needing SQL analytics on big data.

55. Partitions and Bucketing in Hive

🔥 Very Important Long Question

Partitioning

Partitioning divides a Hive table into parts (sub-directories) based on the values of one or more columns. Each unique partition value creates a subdirectory in HDFS. Queries filter on partition columns skip irrelevant directories entirely.

Example: Partition sales data by year and month:

-- HDFS directory structure:
/warehouse/sales/year=2024/month=01/data.orc
/warehouse/sales/year=2024/month=02/data.orc
/warehouse/sales/year=2023/month=12/data.orc

-- Query only reads year=2024/month=01 — skips all others
SELECT SUM(amount) FROM sales WHERE year=2024 AND month=1;

Types of Partitioning

  • Static Partitioning: User specifies partition value manually in LOAD/INSERT.
  • Dynamic Partitioning: Partition values determined automatically from data during INSERT.

Bucketing

Bucketing divides data within a partition into a fixed number of files (buckets) based on the hash of a column value. CLUSTERED BY (col) INTO N BUCKETS. 
CREATE TABLE employees_bucketed (
  name STRING, salary DOUBLE
)
CLUSTERED BY (emp_id) INTO 32 BUCKETS
STORED AS ORC;

Partitioning vs Bucketing

📦 MODULE 2: APACHE SPARK

58. Introduction to Apache Spark

Apache Spark is a fast, general-purpose, open-source cluster computing framework designed for large-scale data processing. It provides in-memory computation, which makes it up to 100x faster than Hadoop MapReduce for iterative algorithms.

Spark was developed at UC Berkeley AMPLab in 2009 and donated to Apache in 2013. It overcomes MapReduce's limitation of writing intermediate results to disk after every step by keeping data in memory (RAM) as RDDs.

59. Features of Apache Spark

  • Speed: 100x faster than Hadoop MapReduce in memory; 10x faster on disk (due to fewer disk I/O operations).
  • Ease of Use: APIs in Python (PySpark), Scala, Java, R. Interactive shell.
  • Generality: Supports batch processing, streaming, SQL, ML, and graph processing in a unified framework.
  • Fault Tolerance: RDD lineage allows automatic re-computation of lost data.
  • Lazy Evaluation: Transformations are not executed until an action is called — allows optimization.
  • In-Memory Computing: Data is cached in RAM between operations (unlike MapReduce's HDFS I/O).
  • Compatibility: Runs on Hadoop (YARN), standalone, Kubernetes, Mesos. Reads from HDFS, S3, Cassandra, HBase.

61. RDD (Resilient Distributed Dataset)

🔥 Very Very Important
RDD (Resilient Distributed Dataset) is the fundamental data abstraction in Spark. It is an immutable, distributed collection of objects that can be processed in parallel across a cluster. RDDs are fault-tolerant through lineage (if a partition is lost, it can be recomputed from its parent RDDs).

Properties of RDD

  • Resilient: Fault-tolerant via lineage — lost partitions are recomputed automatically.
  • Distributed: Data is spread across multiple nodes in the cluster.
  • Dataset: Collection of data records/elements.
  • Immutable: Once created, cannot be modified. Only new RDDs can be derived.
  • Lazy Evaluation: Transformations are not executed until an action triggers execution.
  • Partitioned: Data is split into partitions processed in parallel.
  • In-Memory: Can be cached/persisted in RAM for fast access.

Creating RDDs

from pyspark import SparkContext
sc = SparkContext("local", "MyApp")

# Method 1: Parallelize a collection
rdd1 = sc.parallelize([1, 2, 3, 4, 5])

# Method 2: From a file (HDFS/local)
rdd2 = sc.textFile("hdfs://data/file.txt")

# Method 3: From another RDD (transformation)
rdd3 = rdd1.map(lambda x: x * 2)

63. Word Count using Apache Spark (Python)

from pyspark import SparkContext

# Initialize SparkContext
sc = SparkContext("local", "WordCountApp")

# Step 1: Load input file
lines = sc.textFile("hdfs://input/data.txt")

# Step 2: Split each line into words (flatMap)
words = lines.flatMap(lambda line: line.split(" "))

# Step 3: Create (word, 1) pairs
pairs = words.map(lambda word: (word, 1))

# Step 4: Count each word (reduceByKey is like Combiner+Reducer)
word_counts = pairs.reduceByKey(lambda a, b: a + b)

# Step 5: Sort by count descending (optional)
sorted_counts = word_counts.sortBy(lambda x: x[1], ascending=False)

# Step 6: Save result
sorted_counts.saveAsTextFile("hdfs://output/wordcount")

# Or display top 10
print(sorted_counts.take(10))

sc.stop()

64. Spark Architecture

🔥 Important
Spark Architecture — Driver & Executors

Spark Execution Flow

  1. User writes Spark code with transformations and actions.
  2. SparkContext creates a DAG (Directed Acyclic Graph) of operations.
  3. DAG Scheduler splits DAG into Stages (based on shuffle boundaries).
  4. Task Scheduler sends tasks to Executors on worker nodes.
  5. Executors run tasks, cache intermediate RDDs in RAM, return results to driver.
Spark TermMeaning
JobComplete computation triggered by an action (e.g., collect())
StageSet of tasks within a job that can run without a shuffle. Stages separated by shuffle operations.
TaskA single unit of work sent to one Executor. One task per partition.
ExecutorJVM process on a worker node that runs tasks and caches data.
PartitionA chunk of data in an RDD. Tasks process one partition each in parallel.
